Deployment and performance

Deployment options

By default ICS is installed into a default directory, allowing you to configure the settings to your requirements before deploying.

You may wish to install ICS on a standalone server that is separate from the MCS instance because the transcoding workload is very CPU-intensive. To run ICS as a stand-alone application you should rename the ics-web.xml file to web.xml. If ICS is to be run with another application, then you should copy the servlet element and all its content from ics-web.xml into the web.xml file where the other application is installed.

<servlet>
   <servlet-name>ICS</servlet-name>
   <servlet-class>com.volantis.transcoder.Server</servlet-class>
   <init-param>
      <param-name>config.file</param-name>
      <param-value>[path]\MCS\ics\webapps\ics\WEB-INF\ics-config.xml
      </param-value>
   </init-param>
</servlet>

If the server is to run in a clustered environment, you should ensure that the ics-config.xml file can be accessed from all servers by entering the correct path in the param-value element.

Accelerated performance

The ICS installer includes the Java Advanced Imaging (JAI) core libraries jai_codec.jar and jai_core.jar, located in ics/webapps/ics/WEB-INF/lib. These libraries provide all the basic image transcoding functions, but are less performant than native language acceleration libraries that are available for some platforms. If a native implementation is present, JAI will choose it for any available function.

To improve transcoding you can download the native language libraries for your platform from Java Advance Imaging API.

If MCS is deployed on an platform for which JAI native acceleration libraries are not available, it may be advisable to host ICS on a platform that can support these libraries.
